Design Review — Loading + Empty States
Skeletons
+ empty states
Three screens that currently show a spinner instead of skeleton content, plus the two empty states that bypass the shared EmptyState component. Closing the last visual consistency gap.
01 — Skeleton

1
Shape matches real card exactly
Skeleton thumbnail is 58×58px with 12px radius — identical to the real CommunityCard thumbnail. Text lines are at the same widths/heights as the real title, subtitle, and member-count rows. No layout shift on load.
2
Fade out the last 1–2 cards
Bottom skeleton cards fade to lower opacity (0.6 → 0.4) to suggest more content below. Communicates scroll depth without overclaiming how many results will load.
3
Min. display: 300ms even on fast load
Use Promise.all([fetch, delay(300)]) to prevent a flash of skeleton on fast connections. Flash-of-skeleton is more jarring than a brief hold.

1
Inputs stay visible during search
The city, radius, and categories remain rendered with their real values while results load. Users can see what they searched without a full-screen skeleton wash. Only the results section goes into skeleton state.
2
Search button → loading spinner
The "Search" CTA mutes to bgTer and gains a circular border spinner + "Searching…" label. Uses the same teal border-top trick as all other loading indicators in the app.
3
Show 3 skeleton cards, fade last
Always show exactly 3 skeleton result cards regardless of expected count. Third card fades to 0.5 opacity. Avoids the jarring jump from "1 skeleton" to "7 results" when counts differ.
04 — Skeleton
Gem Swipe Loading
9:41
Gem Swipe

1
3-card stacked depth effect
Three skeleton cards in a CSS transform stack (scale 0.9/0.95/1.0 + translateY) mirror the real deck's stacked appearance. No visual jump between skeleton and the first real card revealing itself.
2
Staggered shimmer on each card
Front card: delay 0.4s. Middle: 0.2s. Back: 0s. The offset shimmer waves give the stack a sense of depth and motion without any JS.
3
Action buttons shown as muted skeletons
The X/❤/⭐ action circles are visible but shimmer-greyed at low opacity. Layout is identical to the loaded state — no shift when the real buttons appear. Middle button is larger (64px vs 52px) matching the real heart CTA.

1
Compass with checkmark — not a sad icon
The compass gets a teal checkmark replacing its centre dot — "mission complete" framing. The coral N-needle stays, faded. Same motif as every other screen, same brand language. Avoid empty-box or magnifying-glass clichés.
2
Primary CTA: expand radius, not refresh
"Expand to 50 km" jumps to a pre-set radius increase. Pulling-to-refresh an empty deck yields nothing — expanding the search area adds real value. The current radius is read from state so the CTA always shows the next logical step.
3
"X saved this session" retention chip
Shows how many gems the user saved this session — a positive reinforcement before they leave. sessionSaveCount from the Swipe store. Only shown when count > 0.

1
SE worst case — nothing clipped ✓
iPhone SE: 667pt height, keyboard ~260pt → ~407pt for UI. Logo + title + 2 fields + Sign In button + "Forgot password?" all fit. Key: no decorative tagline or animation on the auth header — stripped to essentials only.
2
secureTextEntry + "Show" toggle
Plain "Show" text toggle (not an icon) avoids a tiny touch target. Password field uses secureTextEntry. Return key = "go" which submits the form via onSubmitEditing.
3
textContentType for Keychain autofill
Email: textContentType="emailAddress". Password: textContentType="password". Sign-up: newPassword. iOS shows Keychain suggestion above keyboard — removes friction entirely.
